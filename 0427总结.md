# 0427总结

## 预约流程总览

当前代码的预约主链路是：

1. `main()` 读取 `config.USERS`，为每个账号启动一个线程。
2. 线程先完成登录，再进入目标校区和目标自习室。
3. 到达抢座阶段后，按 `PREFER_SEATS` 的顺序逐个尝试座位。
4. 每个座位都会经历“选时间 -> 触发提交 -> 识别验证码 -> 点确认 -> 检查结果”。
5. 成功后发送邮件通知；全部失败后结束当前账号任务。

## 两种模式

| 模式 | 开关 | 核心差异 |
| --- | --- | --- |
| 立即抢座 | `WAIT_FOR_0630 = False` | 启动后直接登录、进房间、开始抢座 |
| 预约抢座 | `WAIT_FOR_0630 = True` | 先等待到指定时间，再进入抢座主流程 |

## 立即抢座流程

1. 程序启动后，线程直接创建浏览器并登录。
2. 登录成功后进入目标校区和目标自习室。
3. 按 `PREFER_SEATS` 顺序依次尝试座位。
4. 每个座位先点击座位号，再选择开始/结束时间。
5. 如果弹出点选验证码，就执行：
   - 拉取验证码图片
   - 走 `ttshitu API` 或本地 `ddddocr`
   - 按识别结果依次点击文字
   - 点“确认”
6. 再调用结果检查：
   - 出现“预约成功”或“有效预约” -> 成功
   - 出现“已有预约”或“预约失败” -> 失败，切换下一个座位
7. 全部优先座位失败后，当前账号任务结束。

## 预约抢座流程

预约抢座比立即抢座多了“时间控制”：

1. 线程先等到 `prep_at = fire_at - 60s`。
2. 到点后启动浏览器并完成登录。
3. 提前进入目标自习室，等待到 `pre_fire_at = fire_at - 10s`。
4. 在 `pre_fire_at` 开始预解验证码，提前把验证码图片和坐标准备好。
5. 到 `fire_at` 附近再点“确认”，尽量让真正提交请求打到整点。
6. 第一个成功锁定的座位会消耗定时窗口，后续座位会切到立即模式，不再重复等待。
7. 后续步骤和立即抢座一致，还是“验证码 -> 确认 -> 检查结果 -> 成功/失败切换”。

## 当前实现里的关键点

- 预约座位是严格按 `PREFER_SEATS` 顺序走的，不是随机优先。
- `select_random_available()` 和 `attempt_seat_selection()` 目前没有接入主流程。
- `build_browser_session_plan()` 和 `session_deadline` 这套会话重启逻辑目前也没有真正接到主流程里。
- 成功后会发邮件通知；失败截图会写到 `logs/`。
- 一个账号全部优先座位失败后，会触发全局停止信号，当前账号任务直接结束。

## 一句话总结

立即抢座是“登录后马上冲”，预约抢座是“提前登录和预热，等到 6:30 前后卡点提交”；两者最终都走同一套“选座 -> 验证码 -> 提交 -> 结果判定”的闭环。
